，# 实验报告：基于 ShardingSphere-Proxy 的分布式数据库架构

## 一、实验目的

搭建一个高可用、可扩展的分布式数据库架构。通过引入 ShardingSphere-Proxy 作为透明的数据库代理层，实现底层数据的**读写分离**与基于 `user_id` 的**水平分片（Sharding）**。整个架构采用 Docker Compose 进行容器化编排与部署。

## 二、系统架构

系统整体分为四层，部署在 Docker Compose 自定义网络 `sharding_net` 中：

```
客户端
  │
  ▼
┌──────────┐
│  Nginx   │  负载均衡，对外暴露统一入口 (端口 80)
└────┬─────┘
     │ 轮询分发
┌────┴─────┐
│          │
▼          ▼
┌──────┐ ┌──────┐
│ App1 │ │ App2 │  Spring Boot 无状态应用 (端口 8080)
└──┬───┘ └──┬───┘
   │        │
   └───┬────┘
       ▼
┌──────────────┐
│ ShardingSphere│  数据库代理层 (端口 3317)
│    Proxy      │  解析 SQL，执行路由、分片与读写分离
└──┬───┬───┬───┘
   │   │   │
   ▼   ▼   ▼
┌─────┐┌─────┐┌─────┐
│主库 ││从库0││从库1│  MySQL 8.0 (主从复制)
│:3306││:3307││:3308│
└─────┘└─────┘└─────┘
```

- **接入层（负载均衡）：** Nginx 节点，将流量分发至后端两个 Spring Boot 实例
- **应用层（业务逻辑）：** 2 个 Spring Boot 实例，仅需配置连接至 Proxy，无需感知底层物理库
- **代理层（中间件）：** ShardingSphere-Proxy 5.5.1，透明的数据库代理
- **存储层（物理库）：** MySQL 8.0 集群（1主2从），通过 Binlog 实现主从同步

### 4.2 存储层：MySQL 主从集群

#### 主库初始化脚本 `docker/mysql/master/init.sql`

在主库中创建数据库 `user_db` 和两张分片物理表 `t_user_0`、`t_user_1`，并创建复制账号。

```sql
CREATE DATABASE IF NOT EXISTS user_db;
USE user_db;

CREATE TABLE IF NOT EXISTS t_user_0 (
    user_id BIGINT NOT NULL PRIMARY KEY,
    username VARCHAR(64) NOT NULL,
    email VARCHAR(128),
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS t_user_1 (
    user_id BIGINT NOT NULL PRIMARY KEY,
    username VARCHAR(64) NOT NULL,
    email VARCHAR(128),
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 创建复制账号
CREATE USER IF NOT EXISTS 'repl'@'%' IDENTIFIED WITH mysql_native_password BY 'repl_password';
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'%';
FLUSH PRIVILEGES;
```

#### 从库初始化脚本 `docker/mysql/slave0/init.sql`（slave1 同理）

从库通过 `CHANGE MASTER TO` 指向主库，建立单向同步链路。

```sql
CHANGE MASTER TO
    MASTER_HOST='ds_master',
    MASTER_USER='repl',
    MASTER_PASSWORD='repl_password',
    MASTER_LOG_FILE='mysql-bin.000001',
    MASTER_LOG_POS=0;

START SLAVE;
```

#### MySQL 配置（通过 docker-compose command 参数）

由于 Windows Docker 挂载配置文件存在权限问题（详见问题排查），MySQL 参数通过 `command` 字段直接传入：

- **主库**：`--server-id=1 --log-bin=mysql-bin --binlog-format=ROW`
- **从库0**：`--server-id=2 --relay-log=relay-bin --read-only=1`
- **从库1**：`--server-id=3 --relay-log=relay-bin --read-only=1`

### 4.3 代理层：ShardingSphere-Proxy 配置

#### 全局配置 `docker/sharding-proxy/conf/global.yaml`

```yaml
rules:
  - !AUTHORITY
    users:
      - user: root@%
        password: root
    privilege:
      type: ALL_PRIVILEGES_PERMITTED

props:
  sql-show: true
```

#### 数据库规则配置 `docker/sharding-proxy/conf/database-logic_db.yaml`

此文件是架构核心，定义了数据源、读写分离规则和分片规则。

```yaml
databaseName: logic_db

dataSources:
  ds_master:
    url: jdbc:mysql://ds_master:3306/user_db?serverTimezone=UTC&useSSL=false&allowPublicKeyRetrieval=true
    username: root
    password: password
  ds_slave_0:
    url: jdbc:mysql://ds_slave_0:3306/user_db?serverTimezone=UTC&useSSL=false&allowPublicKeyRetrieval=true
    username: root
    password: password
  ds_slave_1:
    url: jdbc:mysql://ds_slave_1:3306/user_db?serverTimezone=UTC&useSSL=false&allowPublicKeyRetrieval=true
    username: root
    password: password

rules:
  - !READWRITE_SPLITTING
    dataSourceGroups:
      rw_ds:
        writeDataSourceName: ds_master
        readDataSourceNames:
          - ds_slave_0
          - ds_slave_1
        loadBalancerName: round_robin
    loadBalancers:
      round_robin:
        type: ROUND_ROBIN

  - !SHARDING
    tables:
      t_user:
        actualDataNodes: rw_ds.t_user_${0..1}
        tableStrategy:
          standard:
            shardingColumn: user_id
            shardingAlgorithmName: t_user_inline
    shardingAlgorithms:
      t_user_inline:
        type: INLINE
        props:
          algorithm-expression: t_user_${user_id % 2}
```

**配置解读：**

- 定义三个物理数据源 `ds_master`、`ds_slave_0`、`ds_slave_1`
- 读写分离规则：写操作路由到 `ds_master`，读操作以 ROUND_ROBIN 策略在两个从库间负载均衡
- 分片规则：逻辑表 `t_user` 的数据按 `user_id % 2` 分散到 `t_user_0`（偶数）和 `t_user_1`（奇数）

<mapper namespace="org.example.lab2.mapper.UserMapper">

        VALUES (#{userId}, #{username}, #{email}, #{createTime})

package org.example.lab2.service;

import java.time.LocalDateTime;

@Service

    }

    public List<User> findAll() {
        return userMapper.findAll();

}

## 六、知识点归纳

### 6.1 透明化架构（Proxy 模式）

Proxy 模式的最大优势在于对业务代码**零侵入**。应用层将 ShardingSphere-Proxy 视为一个普通的 MySQL 数据库，仅需将 JDBC URL 指向 Proxy 的地址和端口。复杂的 SQL 解析、路由改写和结果归并由 Proxy 在协议层完成。业务代码中直接操作逻辑表 `t_user`，无需编写任何分片逻辑。

### 6.2 逻辑库与物理库映射

| 概念 | 实例 | 说明 |
|------|------|------|
| 逻辑库 | `logic_db` | 业务视角的统一入口，由 Proxy 配置文件中 `databaseName` 定义 |
| 逻辑表 | `t_user` | 业务代码操作的表名 |
| 物理库 | `user_db`（主从各一份） | MySQL 中真实存在的数据库 |
| 物理表 | `t_user_0`、`t_user_1` | MySQL 中真实存储数据的表 |

Proxy 负责：
1. 解析应用层发来的 SQL 语法树（AST）
2. 根据 `user_id % 2` 计算目标物理表名，将 SQL 改写为针对特定物理表的 SQL
3. 将结果归并后返回给应用层

### 6.3 读写分离与分片的嵌套

架构中先定义了读写分离的逻辑数据源 `rw_ds`（主库写 + 两个从库读，ROUND_ROBIN 负载均衡），再在此基础上配置数据分片规则。这保证了：
- 所有**写操作**（INSERT、UPDATE、DELETE）路由到主库
- **读操作**（SELECT）在两个从库间负载均衡
- 分片在读写分离之上工作，逻辑表 `t_user` 的实际数据节点为 `rw_ds.t_user_${0..1}`

### 6.4 MySQL 主从复制原理

- **主库**开启 Binlog（二进制日志），记录所有数据变更操作
- **从库**通过 IO 线程连接主库，拉取 Binlog 并写入本地 Relay Log
- 从库的 SQL 线程读取 Relay Log，重放其中的 SQL 语句，实现数据同步
- 每个实例必须有唯一的 `server-id`

### 6.5 Docker Compose 服务编排

- 使用 `depends_on` + `condition: service_healthy` 确保服务启动顺序
- `healthcheck` 通过 `mysqladmin ping` 和 TCP 端口检查验证服务就绪
- 所有服务加入同一自定义网络 `sharding_net`，通过容器名互相访问

## 七、实验过程中遇到的问题与解决方案

### 问题 1：ShardingSphere-Proxy 5.5.1 配置文件格式不兼容

**现象：** Proxy 启动时报各种 YAML 解析错误，如 `Cannot create property=schemaName`、`Unable to find property 'provider'`、`Unable to find property 'dataSources'` 等。

**原因：** ShardingSphere 5.5.1 版本对配置格式做了大量不兼容变更，而网上大多数教程和示例基于 5.3.x 或 5.4.x 版本。

**具体变更与解决：**

| 变更项 | 旧版本格式 | 5.5.1 正确格式 |
|--------|-----------|---------------|
| 全局配置文件名 | `server.yaml` | `global.yaml` |
| 数据库配置文件名 | `config-*.yaml` | `database-*.yaml` |
| 数据库名称字段 | `schemaName: logic_db` | `databaseName: logic_db` |
| 认证用户格式 | `root@%:root`（字符串） | `user: root@%` + `password: root`（结构化） |
| 权限配置属性 | `provider` | `privilege` |
| 读写分离属性 | `dataSources` | `dataSourceGroups` |

**解决方法：** 查阅 ShardingSphere 5.5.1 官方文档（https://shardingsphere.apache.org/document/5.5.1/en/），逐一修正配置格式。

### 问题 2：同一个写库不能出现在多个读写分离组

**现象：** Proxy 报错 `DuplicateReadwriteSplittingActualDataSourceException: Readwrite-splitting ds_master data source 'WRITE' is duplicated`。

**原因：** 原始设计方案为偶数 user_id 的读请求走从库0、奇数走从库1，因此定义了两个读写分离组 `pr_ds_0` 和 `pr_ds_1`，都使用 `ds_master` 作为写库。但 ShardingSphere 5.5.1 规定同一个物理数据源只能属于一个读写分离组。

**解决方法：** 将两个读写分离组合并为一个 `rw_ds`，包含两个从库，通过 `ROUND_ROBIN` 负载均衡算法分配读请求。分片规则中的 `actualDataNodes` 从 `pr_ds_${0..1}.t_user_${0..1}` 改为 `rw_ds.t_user_${0..1}`。

### 问题 3：ShardingSphere-Proxy 缺少 MySQL JDBC 驱动

**现象：** Proxy 日志无错误但逻辑库未注册，`SHOW DATABASES` 中没有 `logic_db`。

**原因：** ShardingSphere-Proxy 的 Docker 镜像默认不包含 MySQL JDBC 驱动，需要手动下载放入 `ext-lib` 目录。

**解决方法：** 下载 `mysql-connector-j-9.1.0.jar` 放入 `docker/sharding-proxy/ext-lib/`，并在 `docker-compose.yml` 中添加卷挂载。

### 问题 4：MySQL 主从复制失败 —— server-id 冲突

**现象：** 从库报错 `The replica I/O thread stops because source and replica have equal MySQL server ids`。

**原因：** Windows Docker 挂载的文件权限为 777（World-writable），MySQL 出于安全考虑直接忽略这类配置文件（日志提示 `World-writable config file is ignored`），导致主从库均使用默认 server-id=1。

**解决方法：** 放弃通过挂载 `my.cnf` 文件配置 MySQL，改为在 `docker-compose.yml` 的 `command` 字段中直接传递 MySQL 启动参数（如 `--server-id=1 --log-bin=mysql-bin`），绕过文件权限问题。

### 问题 5：MySQL 主从复制认证失败

**现象：** 从库 IO 线程报错 `Authentication plugin 'caching_sha2_password' reported error: Authentication requires secure connection`。

**原因：** MySQL 8.0 默认使用 `caching_sha2_password` 认证插件，该插件要求 SSL 安全连接。从库到主库的复制连接未启用 SSL，导致认证被拒绝。

**解决方法：** 在主库创建复制用户时显式指定 `mysql_native_password` 认证插件：
```sql
CREATE USER IF NOT EXISTS 'repl'@'%' IDENTIFIED WITH mysql_native_password BY 'repl_password';
```

### 问题 6：MyBatis Spring Boot Starter 版本不兼容

**现象：** Spring Boot 启动报错 `No qualifying bean of type 'org.example.lab2.mapper.UserMapper' available`，MyBatis 的 Mapper 接口未被自动扫描注册。

**原因：** 项目使用 Spring Boot 4.0.5，而初始引入的 `mybatis-spring-boot-starter` 版本为 3.0.4，该版本仅兼容 Spring Boot 3.x。

**解决方法：** 升级 `mybatis-spring-boot-starter` 至 4.0.1，该版本兼容 Spring Boot 4.x。MyBatis Spring Boot Starter 的版本与 Spring Boot 主版本严格对应：2.3.x → Boot 2.7，3.x → Boot 3.x，4.x → Boot 4.x。

## 八、实验总结

本实验成功搭建了一个基于 ShardingSphere-Proxy 的分布式数据库架构，实现了以下目标：

1. **读写分离**：写操作路由到 MySQL 主库，读操作在两个从库间负载均衡
2. **水平分片**：用户表按 `user_id % 2` 分散到两张物理表，对应用层完全透明
3. **负载均衡**：Nginx 将请求分发到两个 Spring Boot 实例
4. **容器化部署**：7 个服务通过 Docker Compose 一键编排启动

实验过程中遇到了 ShardingSphere 版本兼容性、MySQL 主从配置、Docker 文件权限、MyBatis 版本匹配等多方面问题，通过查阅官方文档、分析错误日志逐一解决。这些问题的排查过程加深了对分布式数据库中间件、MySQL 复制机制和容器化部署的理解。
