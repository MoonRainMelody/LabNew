**实验名称**：负载均衡与应用集群
**实验时间**：2026年3月30日
**实验环境**：Windows 11, Docker Desktop, Docker Compose

---

## 一、实验目的

1. **理解负载均衡原理**：通过 Nginx 实现 HTTP 请求的分发，掌握 upstream 的配置方法与轮询策略。
2. **掌握应用集群化部署**：体会 Spring Boot 应用如何通过"去中心化"实现水平扩展，提升系统吞吐量。
3. **学习容器化技术**：利用 Docker 隔离运行环境，模拟多机器部署的真实场景，理解容器编排的优势。
4. **掌握数据库共享**：理解多实例应用如何共享同一个数据库资源，实现数据一致性。

## 二、实验内容概述

本实验构建了一个完整的三层架构系统，模拟电商系统中商品信息查询场景：

- **数据层**：MySQL 8.0 数据库，存储商品信息
- **应用层**：两个完全相同的 Spring Boot 应用实例，提供 RESTful API

## 三、知识点总结

### 3.1 Nginx 负载均衡

**核心概念**：
- **upstream 块**：定义后端服务器组，支持多种负载均衡算法
- **轮询策略（Round Robin）**：默认策略，按时间顺序依次分配请求
- **权重分配（Weight）**：根据服务器性能设置权重，权重越高分配的请求越多
- **健康检查**：通过 `max_fails` 和 `fail_timeout` 实现故障自动剔除

**配置示例**：
```nginx
upstream backend_servers {
    server app1:8080 weight=2 max_fails=2 fail_timeout=10s;
    server app2:8080 weight=1 max_fails=2 fail_timeout=10s;
}
```

**参数说明**：
- `weight=2`：app1 的权重是 app2 的 2 倍，应处理约 67% 的请求
- `max_fails=2`：连续失败 2 次后标记为不可用
- `fail_timeout=10s`：失败后 10 秒内不再尝试连接

### 3.2 Spring Boot 水平扩展

**去中心化架构**：
- 每个应用实例完全独立，不保存状态（无状态设计）
- 所有实例共享同一个数据库，数据集中存储
- 通过负载均衡器统一入口，对客户端透明

**水平扩展优势**：
- 增加实例数量即可提升系统处理能力
- 单个实例故障不影响整体服务可用性
- 可根据负载动态调整实例数量

### 3.3 Docker 容器化技术

**容器 vs 虚拟机**：
- 容器共享宿主机内核，启动更快（秒级）
- 镜像分层存储，节省磁盘空间
- 环境隔离，保证"一次构建，到处运行"

**多阶段构建（Multi-stage Build）**：
- 构建阶段：使用 maven 镜像编译打包
- 运行阶段：使用精简的 jre 镜像运行应用
- 优势：最终镜像体积小，不包含构建工具

### 3.4 Docker Compose 编排

**核心概念**：
- 使用 YAML 文件定义多服务应用
- 自动创建网络，实现服务间通信
- 支持依赖管理（depends_on）和健康检查（healthcheck）

**网络通信**：
- Docker 内置 DNS，服务名直接解析为 IP
- 服务间通过服务名互相访问（如 `mysql:3306`）
- 隔离的容器网络，安全且便于管理

### 4.3 应用服务层实现

**实体类 `Product.java`**：

```java
@Entity
@Table(name = "product")
public class Product {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "name", nullable = false)
    private String name;

    @Column(name = "price", nullable = false, precision = 10, scale = 2)
    private BigDecimal price;

    // 构造方法、getter 和 setter
}
```

**数据访问层 `ProductRepository.java`**：

```java
@Repository
public interface ProductRepository extends JpaRepository<Product, Long> {
}
```

**业务逻辑层 `ProductService.java`**：

```java
@Service
public class ProductService {
    @Autowired
    private ProductRepository productRepository;

    public List<Product> getAllProducts() {
        return productRepository.findAll();
    }
}
```

**控制器 `ProductController.java`**（核心实现）：

```java
@RestController
@RequestMapping("/api")
public class ProductController {
    @Autowired
    private ProductService productService;

    @Autowired
    private Environment environment;

    @GetMapping("/products")
    public Map<String, Object> getProducts() {
        List<Product> products = productService.getAllProducts();
        String port = environment.getProperty("local.server.port");
        String instanceId = environment.getProperty("INSTANCE_ID", "unknown");

        Map<String, Object> response = new HashMap<>();
        response.put("server_port", port != null ? port : "unknown");
        response.put("instance_id", instanceId);  // 关键：标识请求由哪个实例处理
        response.put("data", products);
        return response;
    }
}
```

**实现要点**：
- 返回 `instance_id` 字段用于区分不同的应用实例
- 通过环境变量 `INSTANCE_ID` 注入实例标识，在 docker-compose 中配置
- 返回格式包含 `server_port` 和 `instance_id` 两个标识字段

**配置文件 `application.yml`**：

```yaml
spring:
  application:
    name: lab1
  datasource:
    url: ${SPRING_DATASOURCE_URL:jdbc:mysql://localhost:3306/mall_db?useSSL=false&allowPublicKeyRetrieval=true}
    username: ${SPRING_DATASOURCE_USERNAME:root}
    password: ${SPRING_DATASOURCE_PASSWORD:root}
```

**设计考虑**：

- `ddl-auto: none` 防止 JPA 自动修改数据库结构
- 提供默认值，支持本地开发环境

### 4.5 Nginx 配置

**创建 `nginx/nginx.conf`**：

```nginx
events {
    worker_connections 1024;
}

http {
    upstream backend_servers {
        # 加权轮询，模拟服务器性能差异
        # 健康检查：失败2次后剔除，10秒后重试
        server app1:8080 weight=2 max_fails=2 fail_timeout=10s;
        server app2:8080 weight=1 max_fails=2 fail_timeout=10s;
    }

    server {
        listen 80;

        location / {
            proxy_pass http://backend_servers;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
        }
    }
}
```

**配置说明**：
- `weight=2/1`：app1 和 app2 的请求分配比例为 2:1
- `max_fails=2 fail_timeout=10s`：实现健康检查机制
- `proxy_set_header`：传递真实客户端 IP 到后端

### 4.6 容器编排

**创建 `docker-compose.yml`**：

```yaml
version: '3.8'

services:
  mysql:
    image: mysql:8.0
    container_name: mysql
    environment:
      MYSQL_ROOT_PASSWORD: root
      MYSQL_DATABASE: mall_db
    volumes:
      - ./mysql/init.sql:/docker-entrypoint-initdb.d/init.sql
    ports:
      - "3306:3306"
    networks:
      - app-network
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-uroot", "-proot"]
      interval: 10s
      timeout: 5s
      retries: 5

  app1:
    build: .
    container_name: app1
    environment:
      - SPRING_DATASOURCE_URL=jdbc:mysql://mysql:3306/mall_db?useSSL=false&allowPublicKeyRetrieval=true
      - SPRING_DATASOURCE_USERNAME=root
      - SPRING_DATASOURCE_PASSWORD=root
      - SERVER_PORT=8080
      - INSTANCE_ID=app-instance-1
    depends_on:
      mysql:
        condition: service_healthy
    networks:
      - app-network

  app2:
    build: .
    container_name: app2
    environment:
      - SPRING_DATASOURCE_URL=jdbc:mysql://mysql:3306/mall_db?useSSL=false&allowPublicKeyRetrieval=true
      - SPRING_DATASOURCE_USERNAME=root
      - SPRING_DATASOURCE_PASSWORD=root
      - SERVER_PORT=8080
      - INSTANCE_ID=app-instance-2
    depends_on:
      mysql:
        condition: service_healthy
    networks:
      - app-network

  nginx:
    image: nginx:latest
    container_name: nginx
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf
    ports:
      - "80:80"
    depends_on:
      - app1
      - app2
    networks:
      - app-network

networks:
  app-network:
    driver: bridge
```

**关键配置**：
- `depends_on` + `healthcheck`：确保 MySQL 完全启动后再启动应用
- `INSTANCE_ID` 环境变量：为每个实例注入唯一标识
- 自定义网络 `app-network`：实现服务间通过服务名互相访问

## 六、实验过程中遇到的问题与解决方案

### 6.1 问题一：中文字符编码乱码

**问题描述**：
API 返回的 JSON 中，中文字符显示为 Unicode 编码（如 `\u5fd9\u8259\u6f5e`）

**原因分析**：
- Spring Boot 默认使用 `HttpMessageConverter` 处理 JSON
- MySQL 字符集设置正确（utf8mb4），但序列化时可能使用 ISO-8859-1

**解决方案**：
在 `application.yml` 中添加编码配置：

```yaml
spring:
  http:
    encoding:
      charset: UTF-8
      enabled: true
      force: true
  jackson:
    default-property-inclusion: non_null
```

**验证结果**：中文字符正常显示 ✅

### 6.2 问题二：实例标识不明显

**问题描述**：
最初只使用 `server_port` 字段，两个实例都配置为 8080 端口，无法区分

**解决方案**：
1. 在响应中增加 `instance_id` 字段
2. 通过环境变量 `INSTANCE_ID` 注入唯一标识
3. docker-compose.yml 中配置：
   ```yaml
   environment:
     - INSTANCE_ID=app-instance-1  # app1
     - INSTANCE_ID=app-instance-2  # app2
   ```

**效果**：可以清楚看到每个请求由哪个实例处理

### 6.3 问题三：应用启动时数据库未就绪

**问题描述**：
应用启动失败，提示 "Communications link failure"

**原因分析**：
- Spring Boot 启动速度快于 MySQL 初始化速度
- `depends_on` 只保证启动顺序，不保证服务就绪

**解决方案**：
1. 为 MySQL 配置健康检查
2. 使用 `condition: service_healthy` 等待数据库完全启动

```yaml
mysql:
  healthcheck:
    test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-uroot", "-proot"]
    interval: 10s
    timeout: 5s
    retries: 5

app1:
  depends_on:
    mysql:
      condition: service_healthy
```

**验证结果**：应用稳定启动，无连接失败 ✅

### 6.4 问题四：Nginx 无法解析容器名

**问题描述**：
Nginx 启动报错 "host not found in upstream"

**原因分析**：
- nginx 服务启动时，app1 和 app2 容器还未创建
- DNS 解析失败

**解决方案**：
使用 Docker 的内部 DNS，容器启动后自动解析服务名：
```nginx
upstream backend_servers {
    server app1:8080;  # 使用服务名而非 IP
    server app2:8080;
}
```

同时确保所有服务在同一个网络中：
```yaml
networks:
  app-network:
    driver: bridge
```

### 6.5 问题五：Windows 端口占用

**问题描述**：
启动失败，提示 "port is already allocated"

**解决方案**：
1. 检查端口占用：`netstat -ano | findstr :80`
2. 停止占用端口的进程或修改 docker-compose.yml 中的端口映射
3. 若是 IIS 占用 80 端口，可改为：
   ```yaml
   ports:
     - "8080:80"  # 将 Nginx 映射到 8080 端口
   ```
